# 鳥獣保護管理法様式 ↔ フォーム 双方向ブリッジ（choju_yoshiki）

フォーム「**鳥獣保護管理法許可申請**」と、札幌市の様式ワークブック（**新 7 シート**）の間を
**双方向**でつなぐスタンドアロン Google Apps Script ウェブアプリです。本体アプリ（`gas/`）には一切手を入れません。

- **書き出し（フォーム → Excel）**: レコード詳細の **外部アクション** ボタンが `payload`(JSON) を POST すると、
  様式テンプレートを複製して全シートに**リテラル値**を書き込み、生成スプレッドシートへのリンクを返します。
- **取り込み（Excel → スプレッドシート）**: 本体の**検索画面の外部アクションボタン**（管理者用）から起動します。
  ボタンを押すとこのアプリの取り込み画面が新しいタブで開き、**申請者区分（個人/法人）をラジオで選び**、記入済み
  xlsx を選ぶと解析結果を **分かりやすい表＋チェックボックス**で確認できます。桃（確認用）セルの値も
  **「確認用」メッセージの子質問**として取り込まれ、表に表示されます。取り込みたい申請・従事者を選んで実行すると、
  **親フォーム（申請書）と子フォーム（従事者名簿）のデータ保存用スプレッドシートへ直接 1 行ずつ追記**し、
  あわせて**取り込んだ Excel を親レコードのアップロード欄へ添付**します。本体 API は呼ばず、シートを ID で開いて
  本体と同じレイアウト（ヘッダ 11 行 / データ 12 行目開始 / 固定メタ列 / 子は `pid` 刻印）で書き込む**自己完結方式**です。
  旧来の `mode=import`（`uploadRecords` JSON を返して手で `sync_records` に貼る）も後方互換で残しています。

実体は単一ファイル `Combined.gs`（バナーコメントで `Code/payload/cellmap/domain/fillExport/parseImport/setup/Test` に区切り）。

## 1. 色＝役割の契約（設計の核）

様式 **申請書 L6/L7/L8** にスウォッチ付きの凡例があり、これが読み取り/書き出しの契約です。
色は**設計時の注釈**で、実行時は固定セル番地（`cellmap.gs` セクション）を引きます。

| 色 | RGB | 凡例 | 取り込み | 書き出し |
| --- | --- | --- | --- | --- |
| **黄** | `FFFFFF00` | 正として吸い取る | **権威**として読む | リテラルを書く |
| **桃** | `FFEAD1DC` | 照合・集計確認用 | 集計系（種数/方法 等）=**照合＋プレビュー表示のみ（確認用、保存しない）**（両モード）。同定系（住所/氏名 等）=**照合専用**（取り込まない） | リテラルを書く |
| **緑** | `FF00B050` | 掃き出し場所 | **無視**（出力専用） | リテラルを書く |

**数式は一切使いません**。雛形に残る数式（桃の `=SUM/UNIQUE`、緑の `=申請書!…`）は
「どのセルへ何を書くか」の**仕様書**として読み、出力は全部リテラル値です。

## 2. データ構造

- **従事者名簿が唯一のソース・オブ・トゥルース**。申請書の桃セル（申請者個人・種数・方法）は名簿の集計。
- **名簿の幾何**: ヘッダ 1〜4 行、`firstRow=5 / blockHeight=9 / blockCount=10`（行 5〜94）。**1 ブロック = 1 従事者**。
  - 列: `E`許可番号(緑/出力) / `F`住所 `G`氏名 `H`職業 `I`生年月日(黄・9 行結合) /
    `J`種類 `K`頭数 `M`第2種(卵/ノイヌ/アライグマ) `N`第2数量 /
    `P`用具(行ごとに積層) / `Q-T`狩猟免許 / `U-W`狩猟者登録 / `X-Z`銃器 / `AA`備考。`AC:AE`=入力規則の元リスト（無視）。
  - 種数は申請書と同じ**固定オフセット配置**（キジバト=行 0 …）。方法は P 列に積層。
- **個人/法人**: 個人=申請者を名簿の代表従事者から導出（桃は照合＋確認用）。法人=申請書 F6/F8/F10 を**正**として読む。
  - 取り込み時の判定: **取り込み画面の個人/法人ラジオが最優先**（`forcedType`）。未指定時のみ F8 が代表従事者名と異なれば法人と推定。
  - ⚠️ 申請書 **行10/11 の意味は個人/法人で逆**: 個人 = F10 生年月日 / F11 職業、法人 = F10 代表者名 / F11 職業(空)。
    コードは個人/法人で分岐して読むため正しいが、様式を直すときは両様式の整合に注意。
- **確認用（桃の集計）はプレビュー表示専用・保存しない**: 桃セルの集計系（ほか従事者数 J9・捕獲方法 E30・種ごと捕獲数/採取卵数）を
  `Cho_importConfirm_` が **`displayFields`** に積み、取り込みプレビューに `確認用/…` 行として**桃色（`#c2185b`）で表示**するのみ（`confirm:true`・`table.kv tr.confirm` で着色）。
  **レコードには保存しない**（種数・捕獲方法はフォームの substitution 項目が子から自動集計し、桃の不整合は下記の桃整合性チェックが取り込み時に検出するため、確認用の永続化は不要）。
  同定系（住所/氏名/職業/生年月日・証明書住所/氏名）は**取り込まず**、桃整合性チェックで名簿と照合するだけ（食い違いは `pink_inconsistent`）。
- **証明書の取り込みは保存する**: 証明書シート（被害原因の鳥獣・被害者＋申請者以外の住所/氏名・被害発生の時期/区域・被害の内容・捕獲等を行う理由・備考）は
  親フォームの `証明書` message に該当項目があるため、取り込みで **`fields`（保存対象）** に書き込む（プレビューでは通常の黒文字行＝`confirm:false`）。
- **捕獲用具（具体的な葉用具）の確認表示**（`Cho_buildFriendly_`、保存データは変えない表示専用）:
  - 各従事者に **「捕獲用具一覧（確認用）」** 行を出す。トップのカテゴリ（手捕り/わな/銃器…）ではなく、
    `わな/道具の種類`・`網/道具の種類`・`銃器/銃の種類` の葉用具と `手捕り` を平坦化（`Cho_workerTools_`、`CHO_TOOL_ORDER_` 順）。
    様式の名簿 P 列と 1:1 で見比べられる（例: 田中聡 = `手捕り, はこわな, 空気銃, 散弾銃`）。
  - 申請者セクションの「捕獲方法（従事者集計）」「捕獲方法 照合」の表示行は**廃止**。捕獲方法の名簿照合は表示行を出さず、
    食い違いは下記の桃整合性チェックが出す E30 の `pink_inconsistent` issue（抽出レポート）に委ねる。
- **桃セルの整合性チェック**（`Cho_checkPinkConsistency_`、取り込み時に自動実行）:
  - 各従事者の捕獲用具の和集合と申請書 E30（捕獲方法）が一致するか（**順不同で照合**＝`Cho_comparePinkSet_`）。
  - 申請書の鳥獣ごとの捕獲数/採取卵数が、各従事者の合計と一致するか（`Cho_comparePinkNum_`）。
  - **申請者（個人モードのみ）の同定**: 申請書 F8 氏名＋F10 生年月日で従事者一覧から本人を特定し、
    **見つからなければ**「従事者一覧に見つかりません」を `pink_inconsistent`（cell `F8`）で出す。
    **見つかれば**その本人と住所(F6)/職業(F11)/証明書住所(H4)/証明書氏名(H5)を照合（`Cho_comparePinkText_`）。
    F8/F10 が空なら丸ごと skip（個人様式の桃は数式で未再計算時に空＝誤検知回避）。
  - 不一致は `pink_inconsistent` issue としてプレビューに表示。actual が空のセルは誤検知回避で skip。

## 3. 主要セクション（`Combined.gs`）

| セクション | 役割 |
| --- | --- |
| `Code.gs` | `doPost`（誤送信防止プローブ応答 / 外部アクションリレー→`openUrl` / 後方互換 `mode=import`）・`doGet`（取り込み画面）・JSON 応答 |
| `payload.gs` | `/` パス分解・日付/数値/シリアル変換 |

| `parseImport.gs` | リーダ抽象で名簿→子・申請書/証明書→親を復元（純ロジック）。桃セルは `Cho_importConfirm_` で「確認用」へ取り込み。`uploadRecords` 生成と取り込み画面 UI（個人/法人ラジオ） |
| `write.gs` | 親子シートへ**直接書き込み**（本体 `SyncRecords_` 新規行分岐の移植）。`Cho_appendRow_` / `Cho_buildNewRow_` / `Cho_resolveCell_` と Excel 添付（`Cho_writeRecordsDirect_`） |

| `settings.gs` | 書き込みターゲット / 共有シークレットの登録（`Cho_registerWriteTargets`・Script Properties） |
| `relay.gs` | 外部アクションリレー受信（誤送信防止プローブ HMAC・`ctx` キャッシュ・`openUrl` 生成） |

> 番地を直すときは `cellmap.gs` セクションだけ編集すれば済みます。取り込み・書き出しは同じマップを引きます。

## 4. セットアップ（初回のみ）

1. clasp などで本プロジェクトを GAS にプッシュ（`appsscript.json` に **Advanced Drive Service v2** 依存あり。
   取り込みの xlsx→スプレッドシート変換に必要。スコープは spreadsheets / drive / userinfo.email）。
2. GAS エディタで `Cho_authorize` を一度実行し、Drive / Sheets スコープに同意する。
3. GAS エディタで **書き込みターゲットを登録**：
   `Cho_registerWriteTargets("<親フォームのspreadsheetId>", "<子フォームのspreadsheetId>", "<親の添付項目キー>", "<Excel保存フォルダID(任意)>", "Data", "<共有シークレット(任意)>", "<子のシート名(任意)>")`
   - `spreadsheetId` は各フォームの `settings.spreadsheetId`（= データ保存先のスプレッドシート ID）。本体の設定画面／フォーム定義から確認。
   - **親の添付項目キー** = 親フォーム（申請書）の **file（アップロード）型フィールドの生ヘッダ・パス文字列**。親フォームに file 項目が無ければ追加する（無いと添付はスキップして警告になる）。
   - `Excel保存フォルダID` を省略すると My Drive ルートに残す。`sheetName` 省略時は `Data`。
   - **子のシート名** 省略時は親 `sheetName` にフォールバック（従来挙動）。リレーで `storage.childSheetName`（管理者送信時のみ）が来ればそれを優先する。子フォームのデータシート名が親と異なる場合に指定する。
   - 共有シークレットは本体の管理者設定 `NFB_EXT_ACTION_SECRET` を設定している場合のみ必須（**同じ値**を入れる。未設定なら誤送信防止プローブは来ないので空でよい）。
4. **ウェブアプリとしてデプロイ**（アクセス: 全員 / 実行: 自分）。`/exec` URL を控える。
5. 本体アプリの対象フォーム設定 →「外部アクション」→ **検索画面ボタン**に **URL=`<exec>`・adminOnly=ON** で 1 つ追加する。
   - 検索画面でそのボタンを押すと、本体がサーバ間リレーでこのアプリを叩き、返ってきた取り込み画面 URL を新しいタブで開く。
   - 取り込み先の親スプレッドシート・親シート名は、ボタンの `storage`（管理者送信時のみ同梱）から動的に上書きされる。子スプレッドシート ID・子シート名も `storage.childSpreadsheetId` / `storage.childSheetName` で同梱されればそれを優先し（単票・検索一覧の両ボタンで送出）、欠落時は手順 3 の登録値（無ければ親シート名）にフォールバックする。

## 5. どのシートに何が書かれるか（書き出し）

| シート | 役割 | 内容 |
| --- | --- | --- |
| 申請書 | 正準グリッド | 申請者・種数(固定配置)・目的・期間・区域・方法(リテラル)・処置・区分○ |
| 従事者名簿 | マスタ | 従事者ごと 1 ブロック（種数固定配置・用具積層・免許/登録/銃器） |
| 証明書 | 申請事由 | 被害原因の鳥獣○・被害者(自身/以外+住所/氏名)・時期・区域・理由・備考 |
| 許可証 / 振興局宛通知 / 警察宛通知 | 出力(緑) | 申請書セルをリテラル投影。**許可番号・許可年月日・文書番号は空**（許可後に職員が記入） |
| 従事者証 | 出力(緑) | 代表従事者分（許可証番号は空） |

> ルール: フォーム由来の値はリテラルで書き、**許可後に決まる欄は空のまま**残す。

## 8. フォーム側の整合（様式に合わせて修正済み）・要確認

- **免許/登録番号**: フォームの `番号` は「**〇〇第xxxx号**」一本（旧 `番号接頭語` フィールドは全削除）。
  Excel 名簿 `S`/`V` 列の値と**そのまま一致**するため接頭語の分解/結合は不要（`Combined.gs` から撤去済み）。
- **被害の内容**: 親フォームの該当 message・Excel シートとも **`証明書`** に統一。`被害の内容` が追加され、Excel `証明書!E26` と双方向で対応。
- **規則7条の区分**: フォームの選択肢を様式の 11 区分ラベルに揃えたので ○ グリッドと **1:1**（ロッシーなし）。
  `CHO_AREA7_MARK_` 参照。`具体的名称` は 特定猟具使用禁止区域/特定猟具使用制限区域/休猟区/猟区 に保持。
- 数式変換: 取り込みは**常に名簿マスタを直接読む**ので、桃の集計（`TEXTJOIN/UNIQUE` 等）が変換で崩れても影響なし。
- 取り込みは**追記（新規レコード）が既定**。同じ Excel を二重実行すると重複行になる。取り込み画面の「親レコードID」に
  既存 ID を入れると親 ID を固定できる（子は `<親ID>_cN`）が、**行は上書きされず追記される**点に注意（上書きが必要なら
  本体 `SyncRecords_` の既存行 upsert 分岐の移植が要る＝現状スコープ外）。
- 本体の `sheetLastUpdatedAt` は更新しない（別スクリプトのプロパティのため）。本体クライアントは次回の同期でフル再読込し、
  追記された親子レコードを取得する（正しいが軽く重い）。
- 本番投入前に、実際に記入された xlsx で手動 E2E すること:
  検索画面のボタン → 取り込み画面が開く → 表で確認・一部チェックを外す → 取り込み → 親シートに 1 行・子シートに選択分だけ
  `pid` 付きで追記され、`No.`/`createdAt`/`createdBy`/日付列が正しいこと、親レコードに Excel が添付され本体 UI で開けること、
  本体で再同期して親子リンク（`pid`）が成立することを確認。

## 9. コーディング規約

本体 `gas/` に合わせています。`var` + `function name(){}` スタイル、内部ヘルパは末尾アンダースコア、
関数接頭辞 `Cho_`、定数 `CHO_`。`isDisplayed`/構造 HTML は doGet の自前ページのみ（本体の単一 HTML 配信とは別経路）。
